## Codebase Patterns
- MongoDB is the primary DB (not PostgreSQL) — Mongoose in server/db/mongo-storage.ts, Drizzle types in shared/schema.ts
- Storage is imported as singleton: `import { storage } from '../../db/storage'` — mock with `vi.mock('../../db/storage')`
- vitest is configured at project root (vitest.config.ts), run with `npx vitest run <path>`
- Pre-existing TypeScript errors (~459 after US-003) — don't try to fix unrelated ones, only verify you don't add new ones
- Map iteration in TypeScript: use `Array.from(map)` instead of `for...of` on Maps when target < es2015
- IStorage interface is in server/db/storage.ts; MongoStorage must implement all methods declared there
- Occupation CRUD uses OccupationModel (Mongoose) — schema added in US-002
- TotT modules are in server/extensions/tott/ — each exports functions, not classes
- Server starts with: NODE_OPTIONS='--max-old-space-size=4096' NODE_ENV=development tsx -r dotenv/config server/index.ts
- 60 unit tests already pass in server/__tests__/remaining-gaps.test.ts
- tau-prolog is used for Prolog engine (must be in package.json)
- Building lifecycle has TWO implementations that need merging: building-commission-system.ts and building-lifecycle.ts
- For deterministic tests, mock Math.random with Mulberry32 seeded PRNG: `Math.random = mulberry32(seed)`
- Use `vi.hoisted()` to create mock objects referenced inside `vi.mock()` factories — plain `const` declarations are not available at hoist time
- Telemetry routes use dependency injection (createTelemetryRoutes(storage)) — test by passing mock storage directly, no vi.mock needed
- building-commission-system.ts imports from hiring-system, business-system, and prolog-queries for lifecycle functions
- Hi-fi simulation imports 13 TotT modules — all must be mocked: routine, autonomous-behavior, knowledge, social-dynamics, lifecycle, grieving, economics, building-commission, town-events, drama-recognition, appearance, artifact, education
- InsimulRuleCompiler mock must also use `function` keyword (same constructor pattern as PrologManager)
- The merged lifecycle functions have different signatures than the old simplified ones (e.g., constructBuilding takes worldId/lotId/businessId instead of commissionerId/settlementId)
- For simulation integration tests: mock storage singleton via `vi.mock('../db/storage')`, mock Prolog modules, and mock TotT extension modules (lazy-imported via dynamic import)
- PrologManager mock must use `function` keyword (not arrow fn) for `new` to work in vitest
