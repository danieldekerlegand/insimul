# Insimul Playability Audit

> **Goal:** Ensure that all procedurally generated world data (Rules, Actions, Items, Quests) is actually playable within the constraints of the game engine, and that world intelligence is interpretable and playthroughs are properly isolated from world data.

---

## Executive Summary

The current world generation pipeline produces data that is often **too specific, too creative, and disconnected from what the game engine can actually execute**. The LLM is given high-creativity temperatures (0.9+) and loose prompts that let it invent concepts the engine has no handler for. For example:

- **Actions** like "Negotiate a business license at the municipal library" have no game mechanic — there is no negotiation system, no license concept, no municipal bureaucracy.
- **Rules** like `krewe_captain_prerogative` reference predicates (`cede_priority_to`, `enters`, `is_in_public_venue`) that don't exist in the Prolog KB and can't fire.
- **Items** like "Le doublon espagnol de cuir d'alligator" are overly narrative with no corresponding 3D asset or gameplay function.
- **Quests** generated by Gemini can produce objectives the engine validates away, sometimes leaving only a fallback `talk_to_npc`.

The fix is not to remove AI generation, but to **constrain it tightly to what's achievable** — the same way `quest-objective-types.ts` already constrains quest objectives to a canonical set.

## 1. Base Rules Refactoring

### Current State

- **30 original base rules** (migration 010) covering employment, social dynamics, business, life events, etc.
- **8 new grounded base rules** (migration 019) replacing 1,100+ Ensemble rules: `no_combat_in_settlement`, `must_pay_for_items`, `reputation_decay_on_violation`, `merchant_sells_only_own_stock`, `quest_completion_grants_xp`, `conversation_builds_relationship`, `language_use_grants_xp`, `death_triggers_respawn`

### Problems

1. **The 8 migration-019 rules are too few.** They cover only the most basic game mechanics and leave huge gaps in social simulation, economy, NPC behavior, and world dynamics.
2. **The 30 original rules used predicates that don't exist in the engine** (e.g., `has_occupation`, `business_nearby`, `is_retired`) — they're aspirational, not executable.
3. **AI-generated rules (10 per world) are completely unconstrained.** The LLM invents arbitrary predicates (`cede_priority_to`, `is_in_public_venue`, `enters`) that the Prolog engine has no handler for.
4. **No canonical predicate registry** tells the LLM which predicates actually exist and can be evaluated.

### Required Work

- [ ] **Define a canonical predicate registry** — enumerate every predicate the Prolog engine can actually evaluate (character properties, locations, relationships, inventory, status effects, time-of-day, etc.)
- [ ] **Expand base rules to 40-60** covering:
  - **Economy:** price fluctuation, supply/demand, theft consequences, debt, tax/tithe
  - **Social:** greeting norms, gift-giving etiquette, gossip propagation, reputation thresholds for access, faction standing effects
  - **NPC Behavior:** daily schedules, occupation duties, relationship-driven actions, mood effects on dialogue
  - **World Dynamics:** weather effects, time-of-day restrictions, seasonal events, building access hours
  - **Language Learning:** vocabulary exposure triggers, conversation complexity scaling, assessment unlock conditions, spaced repetition intervals
  - **Combat:** aggression rules, flee conditions, ally behavior, loot distribution
  - **Quest:** auto-generation triggers, difficulty scaling by player level, reward scaling
- [ ] **Constrain AI rule generation** to only use predicates from the canonical registry (similar to how `buildObjectiveTypePrompt()` constrains quest objectives)
- [ ] **Add rule validation** — reject any generated rule that references undefined predicates
- [ ] **Tag rules by system** (economy, social, combat, language, world) for easier management

### Predicate Registry (Draft)

Categories of predicates that should exist:

| Category | Example Predicates |
|----------|-------------------|
| Character State | `character(Id)`, `has_status(Id, Status)`, `energy(Id, E)`, `health(Id, H)`, `mood(Id, M)` |
| Location | `at_location(Id, LocId)`, `location_type(LocId, Type)`, `distance(A, B, D)`, `is_indoor(LocId)` |
| Relationships | `relationship(A, B, Score)`, `faction_standing(Id, Faction, Score)`, `knows(A, B)` |
| Inventory | `has_item(Id, ItemId)`, `item_count(Id, ItemId, N)`, `gold(Id, Amount)` |
| Time | `time_of_day(Hour)`, `day_of_week(Day)`, `season(S)`, `timestep(T)` |
| Occupation | `occupation(Id, Job)`, `employed_at(Id, BusinessId)`, `is_merchant(Id)` |
| Language | `knows_word(Id, Word)`, `proficiency(Id, Dim, Level)`, `conversation_count(Id, N)` |
| World | `truth(Subject, Predicate, Object)`, `building_open(BId)`, `weather(W)` |

## 2. Base Actions Refactoring

### Current State

- **14 base actions** (migration 019): `talk_to_npc`, `buy_item`, `sell_item`, `use_item`, `equip_item`, `give_gift`, `attack_enemy`, `enter_building`, `craft_item`, `travel_to_location`, `compliment_npc`, `learn_word`, `point_and_name`, `solve_puzzle`

### Problems

1. **Only 14 actions is far too few** for a rich simulation. Players and NPCs need a much wider action vocabulary.
2. **AI-generated actions are completely disconnected from game mechanics.** The LLM generates narrative actions ("Negotiate a business license") with no corresponding handler in the game engine.
3. **Actions lack proper pre-conditions and post-conditions.** The current Prolog format has a `can_perform` check on energy only. There's no system for:
   - Pre-conditions: being at the right location, having required items, meeting relationship thresholds, time-of-day requirements
   - Post-conditions: adding truths to the world truth table, changing relationship scores, granting items, triggering quest progress, changing NPC state
4. **No mapping from actions to quest objective types.** An action's post-conditions should be able to satisfy quest objectives.

### Required Work

- [ ] **Expand base actions to 50-80** organized by category:
  - **Social:** greet, compliment, insult, gossip, ask_favor, apologize, invite, flirt, argue, console, thank, introduce_to
  - **Commerce:** buy, sell, haggle, browse_shop, appraise_item, place_order, pay_debt, barter
  - **Exploration:** enter_building, exit_building, travel_to, examine_object, read_sign, search_area, climb, swim, open_door, open_chest
  - **Combat:** attack, defend, dodge, flee, use_ability, heal_self, heal_other
  - **Inventory:** pick_up, drop, use_item, equip, unequip, give_gift, combine_items, craft
  - **Language:** speak_in_target_lang, ask_for_translation, read_aloud, listen_and_repeat, describe_scene, order_food, ask_directions, introduce_self, haggle_in_target_lang
  - **Knowledge:** examine_object, read_sign, study, ask_about_topic, eavesdrop, observe
  - **Work:** work_shift, open_shop, close_shop, restock, tend_garden, cook, clean, repair
  - **Rest:** sleep, eat, drink, sit, meditate, play_music
- [ ] **Define pre-condition/post-condition schema** for every action:
  ```
  action(buy_item) :-
    preconditions: [at_location(Actor, Shop), is_merchant(Target), has_gold(Actor, Price)]
    postconditions: [add_item(Actor, Item), remove_gold(Actor, Price), add_gold(Target, Price), add_truth(bought(Actor, Item, Timestep))]
  ```
- [ ] **Map actions to quest objective completions** — when `buy_item` completes, it can satisfy `collect_item` quest objectives
- [ ] **Constrain AI action generation** to compose from the canonical action set rather than inventing new ones
- [ ] **Add action validation** — reject any generated action that references mechanics the engine can't handle

## 3. Base Items Refactoring

### Current State

- **~50 seed items** (migration 011) across 5 world types (medieval-fantasy, cyberpunk, sci-fi-space, western, universal)
- Expanded taxonomy (migrations 016, 018) with `category`, `material`, `baseType`, `rarity`, `objectRole`
- Merchant-specific inventory templates (bakery, bar, restaurant, shop, grocery, pharmacy, jewelry, brewery, farm)

### Problems

1. **AI-generated items are narrative objects with no gameplay function.** "Le doublon espagnol de cuir d'alligator" can't be used, equipped, or interacted with meaningfully.
2. **No 3D asset mapping for generated items.** The `objectRole` field maps items to asset collections, but AI-generated items get no `objectRole`, so they have no visual representation in the 3D world.
3. **Language learning data on items is often absurd.** Pronunciation guides for complex multi-word French phrases are impractical vocabulary targets. Items should teach single, common words.
4. **Item effects are rarely defined.** Most generated items have no `effects` object, making them functionally inert.
5. **Merchant inventories are hardcoded in English** regardless of world language/setting.

### Required Work

- [ ] **Expand base items to 150-200** with guaranteed `objectRole` mappings, organized by:
  - **Weapons:** sword, dagger, bow, staff, axe, hammer, spear (each with material variants: wood, iron, steel)
  - **Armor:** helmet, chestplate, boots, gloves, shield (leather, iron, steel variants)
  - **Consumables:** health potion, energy potion, antidote, food items (bread, apple, cheese, meat, fish, soup, pie)
  - **Drinks:** water, ale, wine, milk, tea, juice
  - **Tools:** torch, pickaxe, rope, lantern, map, compass, fishing rod, shovel, hammer
  - **Materials:** wood, stone, iron ore, gold ore, leather, cloth, herbs, gems
  - **Collectibles:** coins, jewelry, books, artifacts (with actual lore value or quest relevance)
  - **Keys:** generic key types (house key, chest key, cellar key) with functional door/container unlocking
  - **Language Learning:** flashcards, phrasebooks, dictionaries, labeled objects (simple, single-word targets)
- [ ] **Every base item must have:**
  - A valid `objectRole` mapping to an existing 3D asset
  - Defined `effects` (even if empty `{}`) so the engine knows whether it's usable
  - A `baseType` that matches the asset system
  - A single-word or two-word `name` suitable as a vocabulary target
- [ ] **Constrain AI item generation** to use base items as templates, varying only: name flavor text, description, value range, and rarity — not inventing entirely new item concepts
- [ ] **Fix merchant inventories** to be language/setting-aware and draw from the expanded base item set
- [ ] **Add item validation** — reject items without `objectRole` or with names longer than 4 words

## 4. Procedural Generation Refactoring

### Current State

World generation (`POST /api/generate/complete-world`) runs these steps:
1. **Grammars** — Tracery name grammars (8 categories)
2. **Character Truths** — AI-generated backstories (temp 0.95)
3. **Rules** — 10 AI-generated social rules (temp 0.9)
4. **Actions** — 10 AI-generated or grammar-derived actions (temp 0.7-0.9)
5. **Seed Quests** — 20 template-based quests (no AI)
6. **Items** — 15-20 AI-generated or grammar-derived items (temp 0.8-0.9)
7. **Item Translation** — Gemini FLASH translation (temp 0.2)
8. **Merchant Inventories** — Template-based

### Problems

1. **AI temperatures are too high** for generating functional game data. 0.9+ temperatures produce creative but unplayable content.
2. **No validation against engine capabilities.** Rules reference nonexistent predicates, actions have no handlers, items have no assets.
3. **Generation prompts don't communicate engine constraints.** The LLM doesn't know what the game can actually do.
4. **Quest generation (Gemini) is disconnected from seed quests (templates).** The AI quest generator in `quest-generator.ts` gets a loose prompt and often produces objectives that get normalized away.
5. **No feedback loop.** Generated content isn't tested against the engine before being stored.

### Required Work

- [ ] **Lower AI temperatures** for functional data: 0.3-0.5 for rules/actions/items, 0.7 for narrative flavor text only
- [ ] **Inject engine capability constraints into every generation prompt:**
  - For rules: include the canonical predicate registry
  - For actions: include the canonical action types with pre/post-condition templates
  - For items: include the available `objectRole` values and `baseType` categories
  - For quests: already done via `buildObjectiveTypePrompt()` — extend this pattern
- [ ] **Add post-generation validation:**
  - Rules: parse Prolog content, verify all predicates exist in registry
  - Actions: verify action type is canonical, pre/post-conditions reference valid predicates
  - Items: verify `objectRole` is set, `baseType` is valid, name length is reasonable
  - Quests: already done via `validateAndNormalizeObjectives()` — keep as-is
- [ ] **Shift generation strategy from "invent" to "compose":**
  - Instead of asking the LLM to invent new rules, ask it to **select and parameterize** from the base rule templates for the world's setting
  - Instead of inventing new actions, ask it to **flavor-text existing actions** for the world's culture/language
  - Instead of inventing new items, ask it to **create variants** of base items with setting-appropriate names and descriptions
- [ ] **Add a "generation manifest"** that logs what was generated, what was validated, and what was rejected, viewable in World Intelligence

## 5. World Intelligence Refactoring

### Current State

World Intelligence is a tabbed UI with 5 views:
1. **History** — Timeline, manual editor, family tree
2. **Knowledge Base** — Prolog predicates and facts
3. **Simulations** — Active simulation state
4. **Truths** — Narrative/semantic facts
5. **Telemetry** — Performance monitoring

### Problems

1. **The Knowledge Base view dumps raw Prolog** — not interpretable by most users.
2. **No "world health" summary** — no at-a-glance view of whether the world's rules, actions, items, and quests are internally consistent and playable.
3. **Truths are disconnected from gameplay.** Character truths (personality, secrets) are generated but don't visibly affect NPC behavior or quest availability.
4. **No way to see what's broken.** If a rule references an undefined predicate or an action has no handler, there's no indicator.
5. **History view is mostly empty** for new worlds — the two-phase historical simulation (TotT's 140-year compression) is not implemented.

### Required Work

- [ ] **Add a "World Health" dashboard** as the default World Intelligence view showing:
  - Total rules / valid rules / broken rules (with links to fix)
  - Total actions / executable actions / orphaned actions
  - Total items / items with assets / items without assets
  - Total quests / completable quests / stuck quests
  - NPC coverage: how many NPCs have truths, occupations, relationships
- [ ] **Add a "Generation Log"** showing what was generated during world creation, what was validated, what was rejected
- [ ] **Make the Knowledge Base view human-readable** — translate Prolog facts into natural language summaries (e.g., `has_item(jean, sword)` → "Jean has a Sword")
- [ ] **Connect truths to NPC behavior** — truths should appear as dialogue topics, affect quest generation, and influence NPC disposition
- [ ] **Add validation warnings** — highlight rules with undefined predicates, actions without handlers, items without assets

## 6. Playthrough Persistence Refactoring

### Current State

Playthrough persistence uses a **copy-on-write delta architecture**:
- `playthroughs` table — metadata, status, player character
- `playthroughDeltas` table — entity-level changes (create/update/delete with JSONB diffs)
- `playTraces` table — audit trail of every player action
- `reputations` table — per-playthrough faction/NPC standing

### Problems

1. **World truth table mutations during gameplay may leak back to world data.** When an action's post-conditions add truths (e.g., `bought(player, sword, t=5)`), it's unclear whether these go into the world's truth table or the playthrough's delta layer.
2. **No clear boundary for what's "world data" vs. "playthrough data."** NPC relationship scores, inventory changes, quest progress — these should all be strictly playthrough-scoped, but the current code may modify world entities directly during gameplay.
3. **`worldSnapshotVersion` tracking exists but reconciliation is unclear.** If the world creator updates base data (adds NPCs, changes rules), how does an in-progress playthrough handle the divergence?
4. **No playthrough branching or save slots.** Players can't save at a point and create multiple branches.
5. **Action post-conditions (the new truth-table additions from Section 2) need a clear storage target** — they must go into `playthroughDeltas`, not the world truth table.

### Required Work

- [ ] **Audit every gameplay code path** that modifies world entities (characters, items, truths, rules, relationships) and ensure all mutations go through the delta layer
- [ ] **Define the playthrough truth table** — a separate, playthrough-scoped truth store that:
  - Is initialized from the world's base truths at playthrough start
  - Receives all action post-condition truths during gameplay
  - Is queried by the Prolog engine during gameplay (instead of the world truth table)
  - Is stored as deltas, not as modifications to world data
- [ ] **Implement snapshot reconciliation** — when a playthrough loads and the world has been updated:
  - Compare `worldSnapshotVersion` with current world version
  - Show the player what changed (new NPCs, updated rules, etc.)
  - Allow player to accept changes or continue on old snapshot
- [ ] **Add save slots** — allow 3-5 manual save points per playthrough, stored as delta checkpoints
- [ ] **Add playthrough export** — allow exporting a playthrough's state for analysis or sharing

## Implementation Priority

| Priority | Area | Rationale |
|----------|------|-----------|
| **P0** | Quest generation constraints (Section 4) | Quests are the core gameplay loop; uncompletable quests break the game |
| **P0** | Action pre/post-conditions (Section 2) | Actions are how quests get completed; without conditions, nothing connects |
| **P1** | Base actions expansion (Section 2) | More actions = more ways to complete quests = richer gameplay |
| **P1** | Base rules expansion (Section 1) | Rules govern NPC behavior and world consistency |
| **P1** | Playthrough truth table isolation (Section 6) | Without this, gameplay corrupts world data |
| **P2** | Base items expansion (Section 3) | Items need assets to be visible; this depends on asset pipeline |
| **P2** | Generation prompt refactoring (Section 4) | Improves quality of generated content across the board |
| **P3** | World Intelligence dashboard (Section 5) | Quality-of-life for world creators, not blocking gameplay |
| **P3** | Save slots and branching (Section 6) | Nice-to-have, not critical for core loop |
