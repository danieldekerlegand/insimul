### Key Principles

- **In-app Babylon.js game is canonical.** Every game mechanic, procedural algorithm, and data structure is defined first in the in-app game. Exports reproduce that behavior.
- **Intermediate Representation (IR).** An engine-agnostic JSON document sits between the database and all exporters. Generated by `ir-generator.ts`, it contains world geography, entities, game systems, theme, assets, player config, and UI config.
- **Asset Collections are per-engine.** The `AssetCollection` schema stores model/texture references with per-engine overrides.
- **Shared infrastructure.** All exporters share: `ir-generator.ts`, `asset-bundler.ts`, `asset-resolver.ts`, `foliage-generator.ts`, `ai-bundler.ts`, `plugin-bundler.ts`, `telemetry-config.ts`, `export-naming.ts`.

### Bundled Assets (All Engines)

All export engines receive the same base asset bundle:

- **35+ character models** (Vincent player + Quaternius NPC variants) as GLB/FBX with skeletal animations
- **5 audio files** (ambient village, wind, stone footstep, door creak, button click) as MP3
- **Ground textures** (diffuse, normal, heightmap) as JPG/PNG
- **Building models** (KayKit medieval set) as GLTF+BIN
- **Props and markers** (lanterns, treasure chests, quest markers) as GLB

## 2. Export Pipeline Per Engine

### 2.1 In-App Babylon.js Game

The canonical game running inside the Insimul web application.

| Aspect | Details |
| --- | --- |
| **Location** | `client/src/components/3DGame/` (376+ TypeScript files) |
| **Entry point** | `BabylonWorld.tsx` → `BabylonGame.ts` (pure TS, no React deps) |
| **Data loading** | `ApiDataSource` — live API calls to `/api/worlds/{id}/*` with auth |
| **Save system** | Server sync + IndexedDB with 3-way conflict merge (`SaveQueue.ts`, `SaveConflictResolver.ts`) |
| **AI conversations** | gRPC streaming + Gemini API fallback + local AI (Electron) |
| **Voice input** | WebSocket streaming (always-on) + push-to-talk (HTTP) |
| **Assessment** | Full `/api/assessments/*` server endpoints |
| **TTS** | Server-side TTS + browser SpeechSynthesis fallback |
| **Asset paths** | Absolute (`/assets/...`) |

### 2.2 Babylon.js Standalone Export

Copies the actual 3DGame source into a standalone Vite + Electron project.

| Aspect | Details |
| --- | --- |
| **Location** | `server/services/game-export/babylon/` |
| **Key files** | `babylon-exporter-new.ts`, `game-file-copier.ts`, `babylon-scene-generator.ts`, `babylon-data-generator.ts`, `babylon-code-generator.ts`, `babylon-project-generator.ts` |
| **Approach** | Copies ALL 3DGame `.ts`/`.tsx` files (excluding tests, `.d.ts`), rewrites imports for standalone context |
| **Data loading** | `FileDataSource` — static JSON files bundled in `data/` directory |
| **Save system** | localStorage only (no server sync) |
| **AI conversations** | Gemini API (if `API_URL` configured); local AI via Electron IPC (if models bundled) |
| **Voice input** | Web Speech API → Electron Whisper STT → server fallback chain (`speech-recognition.ts`) |
| **Assessment** | Offline mode: bundled content bank + client-side keyword scoring (`offline-content-bank.ts`, `offline-scoring.ts`); falls back to server if available |
| **TTS** | Electron Piper TTS → browser SpeechSynthesis fallback |
| **Asset paths** | Relative (`./assets/...`), rewritten by `game-file-copier.ts` |
| **Packaging** | ZIP with Electron wrapper, Vite build system |

### 2.3 Unity Export

Re-implements all game systems in C# (86 files).

| Aspect | Details |
| --- | --- |
| **Location** | `server/services/game-export/unity/` |

### 2.4 Godot Export

Re-implements all game systems in GDScript (54 files, ~15,300 lines).

| Aspect | Details |
| --- | --- |
| **Location** | `server/services/game-export/godot/` |

### 2.5 Unreal Engine Export

Re-implements all game systems in C++ (64 .h + 47 .cpp files).

| Aspect | Details |
| --- | --- |
| **Location** | `server/services/game-export/unreal/` |
| **Key files** | `unreal-exporter.ts`, `unreal-cpp-generator.ts`, `unreal-datatable-generator.ts`, `unreal-level-generator.ts`, `unreal-project-generator.ts`, `glb-to-fbx.ts`, `asset-converter.ts` |
| **Approach** | Template-based C++ generation with USTRUCTs for DataTable rows |
| **Data loading** | `DataLoader.h/.cpp` parses DataTable JSON from `Content/Data/` |
| **Coordinate system** | Z-up (Babylon Y-up → Unreal Z-up with 100x cm scaling) |
| **Asset pipeline** | GLB→FBX conversion for rigged models (`glb-to-fbx.ts`); assets routed to `Content/` subdirectories |
| **Setup** | `setup.sh` runs C++ build, level generation commandlet, and Python asset import script |
| **Packaging** | ZIP with `.uproject`; supports UE 5.4–5.7 |

## 4. Remaining Divergences

### 4.1 In-App vs Babylon Export Divergences

| Feature | In-App Behavior | Babylon Export Behavior | Impact |
| --- | --- | --- | --- |
| **Data loading** | Live API calls with auth token | Static JSON files from `data/` | No real-time world updates |
| **Save synchronization** | Server sync + 3-way conflict merge | localStorage only | Playthroughs stay local |
| **NPC conversations** | gRPC streaming + Gemini fallback | Gemini API only (no gRPC server) | Slightly slower response delivery |
| **Voice input** | WebSocket always-on + push-to-talk | Web Speech API → Electron Whisper STT → server fallback | Fully functional via 3-tier chain |
| **Assessment content** | LLM-generated passages + questions | Bundled static content bank (3 languages, multiple passages) | Fixed content pool instead of dynamic; adequate for formative assessment |
| **Assessment scoring** | LLM-based semantic evaluation | Client-side keyword/heuristic scoring | Lower accuracy; sufficient for proficiency estimation |
| **TTS** | Server-side Google TTS | Electron Piper TTS → browser SpeechSynthesis | Quality varies by platform |
| **Asset paths** | Absolute (`/assets/...`) | Relative (`./assets/...`) | Transparent to user |

### 4.2 Cross-Engine Technical Divergences

| Feature | Babylon.js | Unity | Godot | Unreal | Resolution Path |
| --- | --- | --- | --- | --- | --- |
| **Skeletal animation blending** | 10 states with crossfade | Exposes Animator params, no `.controller` | Exposes params, no AnimationTree | Exposes AnimInstance vars, no AnimBlueprint | All non-Babylon engines need editor-created animation assets |
| **Building textures** | Per-style wall/roof with UV tiling | Flat colored materials | Material cache, no UV tiling | Flat BasicShapeMaterial | Textures bundled; users create Materials in editor |
| **Terrain-following roads** | Ribbon geometry sampling terrain | Ribbon geometry | Ribbon mesh with tangent edges | Flat cubes above ground | Unreal could use ProceduralMeshComponent ribbon |
| **Grass rendering** | ThinInstance transforms | Instanced rendering | MultiMesh X-pattern planes | UInstancedStaticMeshComponent | Functionally equivalent approaches |
| **NPC mesh merging** | Merged into single mesh | Component-based | Individual MeshInstance3D | Separate components | UE5 static batching mitigates |
| **Coordinate system** | Y-up (OpenGL) | Y-up | Y-up (Godot 4) | Z-up (swap + 100x cm) | Handled by Unreal exporter |
| **Interior NPC population** | NPCs spawn inside | Same | Full (role-based, Label3D) | Furniture only, no NPCs | Unreal could add from DT_Characters |
| **Custom sky shader** | ShaderMaterial dome gradient | Same | bg color matches fog | bg color (no gradient) | Could add procedural sky material |

## 5. Post-Setup Editor Tasks

These must be completed by the user inside the respective engine editor. Required because these assets are binary formats that cannot be generated from code templates.

### 5.1 Unreal Engine

1. **AnimBlueprint** — `UInsimulAnimInstance` exposes `Speed`, `Direction`, `bIsInAir`, `AnimState`. Create AnimBlueprint in editor with Blend Space 1D (idle=0, walk=200, run=600).
2. **Building textures** — Create Material with Texture Sample → Base Color. Replace `BasicShapeMaterial` references.

### 5.2 Unity

1. **Animator Controller** — `CharacterAnimationController.cs` exposes `Speed`, `IsGrounded`, `IsTalking`, `Attack`, `Interact`, `Die`. Create Animator with Blend Tree.
2. **Building textures** — Create Materials referencing bundled textures; assign to `GetSharedMaterial()` cache.

### 5.3 Godot

1. **AnimationTree** — Configure with blend nodes driven by `speed` and `anim_state`.
2. **Building textures** — Set `uv1_scale` on `StandardMaterial3D` for tiling.

## 7. Planned: Migrate 3DGame to shared/game-engine

### 7.1 Motivation

The game's source of truth currently lives in `client/src/components/3DGame/` (376+ TypeScript files). The Babylon.js export pipeline copies these files and rewrites import paths to produce a standalone project — a fragile process that breaks whenever paths or module boundaries change. Moving game logic into `shared/game-engine/` would let both the in-app game and the Babylon exporter import from the same location, eliminating the copy-and-rewrite step entirely and making the codebase easier to reason about.

### 7.2 File Breakdown

| Category | Files | Babylon.js Deps | Browser API Deps | Move Difficulty |
| --- | --- | --- | --- | --- |
| **Pure business logic** (event bus, quest engine, reputation, crafting, volition, scheduling, gamification, action systems, type definitions) | 63 | None | None | Trivial |
| **Rendering & UI** (procedural generators, UI panels, camera, combat visuals, minimap, VR) | 151 | `@babylonjs/core`, `@babylonjs/gui` | None | Low — Babylon.js becomes a `shared/` dependency |
| **Browser-dependent** (DataSource, ConversationClient, audio managers, VR, streaming) | 50+ | Some | `fetch`, `WebSocket`, `localStorage`, `navigator`, `MediaRecorder` | Medium — needs interface abstraction |
| **React wrapper** (`BabylonWorld.tsx`) | 1 | None | None | Stays in `client/` |

### 7.3 Key Blocker: DataSource.ts

`DataSource.ts` is imported by nearly every game system. It has hard dependencies on `fetch` (for API calls) and `localStorage` (for saves). Before the browser-dependent files can move, this needs to be refactored into an `IDataSource` interface with two implementations:

- **ApiDataSource** — for the in-app web game (uses `fetch` with auth tokens, server sync)
- **FileDataSource** — for the standalone Babylon export (reads bundled JSON, saves to `localStorage`)

The interface would live in `shared/game-engine/`, and each implementation would be injected at initialization time by the host environment.

### 7.4 Migration Phases

#### Phase 1 — Pure logic (63 files) → `shared/game-engine/logic/`

- Copy files as-is (no code changes needed)
- Update ~35 import paths in external consumers (server tests, shared tests, client components)
- Update `game-file-copier.ts` to read from new location
- Risk: minimal

#### Phase 2 — Rendering & Babylon files (151 files) → `shared/game-engine/rendering/`

- Add `@babylonjs/*` to shared package dependencies
- Update import paths in external consumers
- Create barrel exports (`index.ts`)
- Risk: low (Babylon.js is a public library, no client-specific coupling)

#### Phase 3 — Refactor DataSource & browser APIs

- Extract `IDataSource` interface
- Create `ApiDataSource` (stays in `client/`) and `FileDataSource` (moves to `shared/`)
- Update all game systems to accept `IDataSource` via constructor/init rather than importing directly
- Apply same pattern to other browser-dependent modules (audio, WebSocket, etc.)
- Risk: medium (touches core save/load path)

#### Phase 4 — Update export pipeline

- Update `game-file-copier.ts` to point at `shared/game-engine/` instead of `client/src/components/3DGame/`
- Simplify or remove import-path rewriting (files are already in shared)
- Test full export → standalone Electron build
- Risk: low

### 7.5 What Stays in `client/`

- `BabylonWorld.tsx` — thin React wrapper (mounts/unmounts BabylonGame)
- `ApiDataSource` implementation — requires React auth context for tokens
- Any future React-specific UI overlays

## 8. Cross-Cutting Concerns

### 8.1 Deterministic Procedural Generation

All algorithms must produce identical results given the same seed. Requires seeded PRNG (same algorithm across JS, C++, C#, GDScript) and documented algorithm specs.

### 8.2 Rule System Portability

All rules converted to unified IR form. Native engines implement rule evaluators. Godot has the most complete client-side Prolog (889 lines); Unreal has a stub.

### 8.3 Asset Pipeline Per Engine

| Aspect | Babylon | Unity | Godot | Unreal |
| --- | --- | --- | --- | --- |
| **Model format** | GLB/GLTF | GLTF/GLB (auto-import) | GLB (native) | GLB→FBX |
| **Asset storage** | `public/assets/` | `Assets/Resources/` | `res://assets/` | `Content/` |
| **AI model paths** | `./assets/ai/` | StreamingAssets | `res://ai/models/` | local paths |

### 8.4 Telemetry

All engines ship per-engine telemetry clients with consistent behavior:

| Engine | Implementation | Fallback |
| --- | --- | --- |
| Babylon | TypeScript (`ClientTelemetryCollector`) | localStorage |
| Unity | C# (`TelemetryManager`) | File-based |
| Godot | GDScript (`telemetry_client.gd`) | File-based |
| Unreal | C++ (`TelemetrySubsystem.h/.cpp`) | File-based |

### 8.5 Export Naming

All exports follow: `{WorldName}{Engine}{AIMode}` (e.g., "LaLouisianeUnityCloud"). Defined in `export-naming.ts`.

### 8.6 Offline Assessment Architecture

The Babylon.js standalone export includes a complete offline assessment pipeline:

1. **Content generation** — `shared/assessment/offline-content-bank.ts` bundles pre-written passages, comprehension questions, and writing prompts for French, Spanish, and English at beginner difficulty. Each question includes an expected-answer rubric (keywords + accept phrases).
2. **Scoring** — `shared/assessment/offline-scoring.ts` provides keyword/heuristic scoring for reading/listening (keyword matching with partial credit) and writing (word count + keyword coverage + sentence structure analysis).
3. **TTS** — Falls back to Electron Piper TTS → browser SpeechSynthesis.
4. **Assessment CRUD** — Logs to localStorage (no server persistence).
5. **Conversation phase** — Already fully client-side (in-game NPC interaction).

The `AssessmentEngine.ts` tries server endpoints first and transparently falls back to offline mode when they're unavailable.

### 8.7 Offline Voice Input Architecture

Voice input in standalone exports uses a 3-tier fallback chain (implemented in `client/src/lib/speech-recognition.ts`):

1. **Web Speech API** — Browser-native `SpeechRecognition` / `webkitSpeechRecognition`. Works in most modern browsers including Electron.
2. **Electron Whisper STT** — If Web Speech API is unavailable, uses Electron IPC to invoke local Whisper model (`window.electronAPI.aiSTT`). Requires AI models bundled in the export.
3. **Server /api/stt** — Final fallback for the in-app web version. Not available in standalone exports.

The `serverSideSTT()` function automatically detects the Electron environment and routes through the IPC bridge before attempting the server endpoint, ensuring voice input works offline.
