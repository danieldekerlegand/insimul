Section 01

The Core Problem

Vibe coding — the practice of describing what you want and having an AI generate all the implementation code — works beautifully for text files. The challenge with game engine project templates is that the scaffolding layer, the files that define the project to the engine itself, combines text-editable configuration with files that either require the engine to be running to generate, depend on binary formats, or contain GUIDs and checksums that must be internally consistent.

The goal here is to understand exactly which files fall into which category for each engine, and therefore what a vibe-coding pipeline would need to do to generate a valid, immediately openable project from scratch.

Key Framing
The problem is not "generate a game" — it's "generate a project that opens without errors." A vibe-coded project template is a skeleton: correct engine settings, a working main scene/level, placeholder scripts, and a sensible folder structure — all without requiring the developer to manually open the engine to bootstrap the project.

What "project files" actually means

For any game engine, the project-level files break down into four layers:

LayerContentsAI-Generatable?
1. Project Descriptor The root file that tells the engine this folder is a project (.uproject, project.godot, etc.) Yes — JSON/text
2. Engine Settings Physics, rendering, input, audio, build settings stored in config files Mostly text
3. Scene / Level Content The initial scene(s) with cameras, lights, etc. Engine-specific format
4. Asset Import State Cached import data, GUID maps, asset databases — generated by the engine on first open Engine must generate

Layers 1 and 2 are largely solvable with careful text generation. Layer 3 is where each engine diverges significantly. Layer 4 is universally "let the engine do it on first launch."

The .uproject File — Fully AI-Writable

The root .uproject file is plain JSON and is the most important file to get right. A minimal valid example for UE5:

{
  "FileVersion": 3,
  "EngineAssociation": "5.4",
  "Category": "",
  "Description": "",
  "Modules": [
    {
      "Name": "MyProject",
      "Type": "Runtime",
      "LoadingPhase": "Default",
      "AdditionalDependencies": [
        "Engine"
      ]
    }
  ],
  "Plugins": [
    {
      "Name": "ModelingToolsEditorMode",
      "Enabled": true,
      "TargetAllowList": ["Editor"]
    }
  ]
}

Remarkably, an even more minimal version — a single-line JSON object with just {"FileVersion": 3} — is technically enough to open in the editor. UE will scaffold the rest of the folder structure itself. This is a huge advantage for programmatic generation.

Config Files — AI-Writable INI

The Config/ folder uses standard INI format with UE-specific section headers. These are entirely text and can be generated precisely by an AI. For example, DefaultEngine.ini:

[/Script/Engine.RendererSettings]
r.DefaultFeature.AutoExposure.ExtendDefaultLuminanceRange=True
r.DynamicGlobalIlluminationMethod=1
r.ReflectionMethod=1

[/Script/Engine.GameMapsSettings]
GameDefaultMap=/Game/Maps/MainLevel
EditorStartupMap=/Game/Maps/MainLevel

The Level / .umap Problem — The Hard Part

Core Challenge
UE levels (.umap files) are binary by default. They store Unreal's internal object graph in a proprietary serialized binary format. An AI cannot directly write a valid .umap file any more than it can write a valid compiled executable.

There are three approaches to this problem:

Approach A: Skip the level file, let the engine create it

The simplest strategy: generate everything except the level file. When the engine opens the project, it can create a default empty level. The DefaultEngine.ini can specify that no map should load on startup, letting the developer create their first level manually. This works but isn't a complete bootstrap — the developer has one manual step.

Approach B: Use the Engine's own CLI to generate it

Unreal provides UnrealEditor-Cmd.exe (and its equivalents) which can be invoked from the command line to run editor operations including level creation. A vibe-coding pipeline could generate all text files, then invoke the engine itself to create and populate a starter level:

# Generate the project skeleton (text files only)
ai_generate_project MyProject/

# Invoke UE command line to compile and create a default level
<UE_ROOT>/Engine/Binaries/Win64/UnrealEditor-Cmd.exe \
  MyProject/MyProject.uproject \
  -run=resavepackages \
  -map=/Game/Maps/MainLevel \
  -buildlighting \
  -AllowCommandletRendering

Approach C: Use a Blueprint-only project with a minimal level stub

For Blueprint-only projects (no C++), there is a workaround: the NewMap commandlet can create a minimal level. Additionally, some community-maintained minimal project skeletons ship a tiny, pre-generated .umap binary that acts as an "empty level" stub — AI generates everything else, this single binary is copied from a known-good minimal template.

C++ Source Files — Fully AI-Writable

The C++ source layer is completely text. The three critical Build System files follow known, repetitive patterns and are ideal for AI generation:

// MyProject.Build.cs — the module definition
using UnrealBuildTool;
public class MyProject : ModuleRules {
  public MyProject(ReadOnlyTargetRules Target) : base(Target) {
    PCHUsage = PCHUsageMode.UseExplicitOrSharedPCHs;
    PublicDependencyModuleNames.AddRange(new string[] {
      "Core", "CoreUObject", "Engine", "InputCore"
    });
  }
}

Asset Files (.uasset) — AI Cannot Write

All content assets (materials, meshes, textures, blueprints) are stored as .uasset binary files. An AI cannot generate these. The practical resolution is to either skip them (empty Content/) or reference engine-provided starter content that the developer can add from within the editor.

UnrealBuildTool — The Good News

After the text files are in place, GenerateProjectFiles.bat (Windows) or GenerateProjectFiles.sh (Linux/Mac) is a CLI tool that generates all IDE project files (Visual Studio .sln, Xcode workspace). This is entirely automatable — it's just a script invocation. The UBT build system then handles compilation from the text source files alone; no binary stubs are needed for the code layer.

UE Summary
Text-writable: .uproject (JSON), Config INI files, C++ source (.h/.cpp), Build.cs Target.cs.
Binary/engine-required: .umap level files, .uasset content assets, IDE project files.
Strategy: AI generates all text files; level creation requires either engine CLI invocation or a stub binary copied from a minimal template.

The Serialization Mode Decision — Critical

Unity has two serialization modes: binary (default, compact, opaque) and Force Text (YAML-based, human-readable, version-control-friendly). This single setting completely determines whether scene and prefab files are AI-writable.

The Key Insight
Any new project bootstrapped for vibe coding MUST use Force Text serialization. This is controlled by ProjectSettings/EditorSettings.asset. When Force Text mode is active, every .unity scene file and .prefab becomes human-readable YAML that an AI can write, read, and modify.

Setting Force Text in EditorSettings.asset:

%YAML 1.1
%TAG !u! tag:unity3d.com,2011:
--- !u!159 &1
EditorSettings:
  m_ObjectHideFlags: 0
  serializedVersion: 9
  m_ExternalVersionControlSupport: Visible Meta Files
  m_SerializationMode: 2       # ← 2 = Force Text (0 = Mixed, 1 = Force Binary)
  m_LineEndingsForNewScripts: 0
  m_DefaultBehaviorMode: 0
  m_PrefabRegularEnvironment: {fileID: 0}
  m_PrefabUIEnvironment: {fileID: 0}
  m_SpritePackerMode: 4
  m_SpritePackerPaddingPower: 1
  m_EtcTextureCompressorBehavior: 1

Unity Scene Files (.unity) in Text Mode

With Force Text enabled, a .unity scene is a multi-document YAML file. Each GameObject, component, camera, and light is its own YAML document separated by ---. The format is Unity's own YAML subset — not standard YAML, but close enough that it's fully text-writable with care.

A minimal working scene with a camera:

%YAML 1.1
%TAG !u! tag:unity3d.com,2011:
--- !u!29 &1
OcclusionCullingSettings:
  m_ObjectHideFlags: 0
  serializedVersion: 2
  m_OcclusionBakeSettings:
    smallestOccluder: 5
    smallestHole: 0.25
    backfaceThreshold: 100
--- !u!104 &2
RenderSettings:
  m_ObjectHideFlags: 0
  m_Fog: 0
  m_AmbientMode: 0
  m_AmbientSkyColor: {r: 0.212, g: 0.227, b: 0.259, a: 1}
--- !u!1 &123456789      # ← File ID (must be unique within file)
GameObject:
  m_ObjectHideFlags: 0
  serializedVersion: 6
  m_Component:
  - component: {fileID: 123456790}  # ← References Transform below
  m_Layer: 0
  m_Name: Main Camera
  m_IsActive: 1
--- !u!4 &123456790      # ← Transform component
Transform:
  m_ObjectHideFlags: 0
  m_GameObject: {fileID: 123456789}
  m_LocalPosition: {x: 0, y: 1, z: -10}
  m_LocalRotation: {x: 0, y: 0, z: 0, w: 1}
  m_LocalScale: {x: 1, y: 1, z: 1}
--- !u!20 &123456791     # ← Camera component
Camera:
  m_ObjectHideFlags: 0
  m_GameObject: {fileID: 123456789}
  m_Enabled: 1
  near clip plane: 0.3
  far clip plane: 1000
  field of view: 60

The GUID / Meta File Problem

GUID Consistency — A Significant Constraint
Every asset in Unity requires a corresponding .meta file with a globally unique GUID. Scene files reference scripts, materials, and other assets by these GUIDs. If an AI generates a scene that references a script at GUID abc123, the script's .meta file must declare exactly that GUID, or Unity will report missing references.

A script meta file looks like:

fileFormatVersion: 2
guid: a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4  # ← Must be a valid UUID4, consistent across files
MonoImporter:
  externalObjects: {}
  serializedVersion: 2
  defaultReferences: []
  executionOrder: 0
  icon: {instanceID: 0}

The practical solution is for an AI to generate all GUIDs upfront before writing any files, maintain a map of asset paths to GUIDs, and use that map consistently across scene files and meta files. Since GUIDs are just random 128-bit hex strings, an AI has no trouble generating them — it just needs to stay internally consistent.

Package Manager — Simple JSON

The Packages/manifest.json file lists all UPM packages. This is standard JSON and trivially writable:

{
  "dependencies": {
    "com.unity.inputsystem": "1.8.1",
    "com.unity.render-pipelines.universal": "17.0.3",
    "com.unity.textmeshpro": "3.0.9"
  }
}

ProjectSettings Files

All files in ProjectSettings/ use the same Unity YAML format. These are well-documented and AI-writable. The most important ones:

FilePurposeDifficulty
ProjectSettings.assetCompany name, version, build identifiers, scripting definesMedium — many fields
InputManager.assetLegacy input axesEasy — repetitive patterns
TagManager.assetTags and layersEasy
QualitySettings.assetGraphics quality levelsMedium
GraphicsSettings.assetRender pipeline asset referencesMedium — references by GUID
Physics2DSettings.asset2D physics defaultsEasy

The Library Folder — Do Not Touch

The Library/ folder is Unity's internal asset import cache. It is massive (often gigabytes), completely binary, and must be generated by Unity on first open. An AI project generator should never attempt to create this. Unity always regenerates it from the Assets/ and Packages/ sources when none is present.

Batch Mode Bootstrap — The Power Tool

Unity supports a powerful -batchmode CLI flag that runs the engine headlessly. A vibe-coding pipeline could generate all text/YAML files, then invoke Unity once in batch mode to execute an Editor script that performs post-generation setup:

Unity.exe \
  -batchmode -nographics -quit \
  -projectPath ./MyProject \
  -executeMethod ProjectBootstrapper.Initialize \
  -logFile ./bootstrap.log

The ProjectBootstrapper.Initialize static C# method (placed in Assets/Editor/ and generated by the AI) can set up everything Unity won't accept from text files alone: baking lightmaps, forcing asset reimport, creating prefabs from scripts, etc.

Unity Summary
Text-writable (with Force Text mode): .unity scenes, .prefab files, .meta files, ProjectSettings/*.asset, manifest.json, all C# scripts.
Binary / engine-required: Library/ cache (always regenerated), binary mode .unity/.prefab files (avoidable), compiled .dll assemblies.
Key constraint: GUID consistency across all .meta files and scene cross-references — must be managed by the AI generator.
Strategy: Force Text mode + systematic GUID generation = fully AI-writable Unity projects. Optionally use batch mode for post-generation tasks.

Scene Files (.tscn) — The Most AI-Friendly Format

Godot's TSCN format is arguably the most AI-friendly scene format of any major game engine. It's a flat, simple, human-readable text format with minimal hidden complexity:

[gd_scene load_steps=3 format=3 uid="uid://bpfg8yjdnp4g2"]

[ext_resource type="Script" path="res://scripts/game_manager.gd" id="1_a2c4e"]

[sub_resource type="Environment" id="Environment_xyz12"]
background_mode = 2
sky = null
ambient_light_source = 2
ambient_light_energy = 0.8

[node name="Main" type="Node3D"]
[node name="Camera3D" type="Camera3D" parent="."]
transform = Transform3D(1, 0, 0, 0, 1, 0, 0, 0, 1, 0, 2, 8)
[node name="DirectionalLight3D" type="DirectionalLight3D" parent="."]
transform = Transform3D(0.7, -0.5, 0.5, 0, 0.7, 0.7, -0.7, -0.5, 0.5, 0, 5, 0)
light_energy = 1.0
shadow_enabled = true
[node name="GameManager" type="Node" parent="."]
script = ExtResource("1_a2c4e")
TSCN Design Philosophy
Only non-default values are written to .tscn files — properties equal to their defaults are omitted entirely. This means a minimal scene for a working starting point can genuinely be a dozen lines. The parser in Godot (resource_format_text.cpp) is open source and well-documented.

Resource Files (.tres) — Also Text

Materials, resources, and other data objects use the TRES format, which follows the same structure as TSCN. An AI can write materials, animation data, and custom resources in this format.

UID Management — A Minor Complication

Godot 4 introduced UIDs for resources (the uid://... strings you see in scene files). These are used for more robust cross-reference resolution than file paths alone. The format is a base62-like encoded 64-bit integer. An AI needs to generate these to be non-conflicting random-looking strings in the correct format:

# UID format: uid://[base62 chars, ~12 chars]
# Example: uid://bpfg8yjdnp4g2
# These don't need to be globally unique across all projects — just within one

The practical approach is to generate random 64-bit values and encode them in Godot's UID alphabet. The UID is not verified against any database — it's generated locally and used as a stable identifier within the project. An AI can safely generate these.

GDScript — Ideal for AI Generation

GDScript is Python-inspired, dynamically typed (though optionally type-hinted), and has minimal boilerplate. It's one of the most AI-writable scripting languages in the game dev ecosystem:

extends Node3D

@export var player_speed: float = 5.0

func _ready() -> void:
    print("Game started")

func _process(delta: float) -> void:
    pass

The .godot/ Cache Folder

Like Unity's Library/, the .godot/ folder is engine-generated and should never be manually created. Godot regenerates it automatically when opening a project where it's absent. For the headless/CI use case, running godot --editor --headless --quit once will force cache generation before any further operations.

Headless CLI — Very Capable

Godot 4 has excellent headless support:

# Initialize project cache without opening UI
godot --editor --headless --quit --path ./MyProject

# Run a tool script headlessly
godot --headless --script res://tools/setup.gd

# Export the project
godot --headless --export-release "Linux/X11" ./build/game.x86_64

The ability to run arbitrary GDScript files headlessly via --script is particularly powerful for a vibe-coding pipeline. The AI can generate a setup.gd tool script that runs post-generation tasks (generating nav meshes, pre-baking data, etc.) and the pipeline invokes it without any GUI interaction.

Godot Summary
Text-writable: project.godot, all .tscn scene files, all .tres resource files, all GDScript .gd files, all C# .cs files, export_presets.cfg.
Engine-required: .godot/ cache folder (auto-generated on first open). Binary .scn/.res files exist but are always optional alternatives to .tscn/.tres.
Strategy: Godot projects can be almost entirely AI-generated with no binary stubs required. UIDs need deterministic generation logic. The headless --script flag enables powerful post-generation scripting.

Section 06

Implementation Strategies

Strategy 1: Pure Text Generation + First-Launch Bootstrap

The simplest approach. Generate all text files; rely on the engine to complete initialization on first open. This is entirely feasible for all three engines and requires no engine installation in the generation environment.

Unreal

Generate .uproject + Config INI + Source C++ files. Omit .umap — engine will prompt to create a new level. On first compile, UnrealBuildTool handles everything.

Unity

Generate Assets/ + ProjectSettings/ YAML with Force Text enabled + manifest.json. On first open, Unity generates Library/ cache. Scenes open correctly if GUIDs are internally consistent.

Godot

Generate project.godot + .tscn scenes + .gd scripts. On first open, Godot generates .godot/ cache. Everything works immediately — the most complete of the three.

Strategy 2: Stub Binary + AI Text Generation

For Unreal specifically, maintain a library of minimal pre-generated binary stubs (an empty level .umap, a few basic .uasset materials) that the AI references by name. The AI generates all text files and copies the appropriate stubs. This achieves a fully functional out-of-box project.

Stub Library Management
The stub .umap files need to be versioned per UE version — a UE5.4 stub may not be valid in UE5.5. The stub library therefore requires maintenance as Unreal releases new versions. For a production pipeline, automate stub generation using Epic's own project templates as sources.

Strategy 3: Headless Engine in CI/CD

The most powerful but most infrastructurally complex approach. Run the target engine in headless mode as part of the generation pipeline:

#!/bin/bash
# Full pipeline with headless engine

# Step 1: AI generates all text files
generate_text_files --config template.yaml --output ./MyProject

# Step 2: Godot headless to initialize cache + run setup
godot --editor --headless --quit --path ./MyProject
godot --headless --script res://tools/post_generate.gd --path ./MyProject

# Step 3: For Unity with batchmode
Unity.exe -batchmode -nographics -quit \
  -projectPath ./MyProject \
  -executeMethod Bootstrap.Initialize

# Step 4: Package as zip for delivery
zip -r MyProject.zip MyProject/ --exclude "*.godot/*" --exclude "Library/*"

Strategy 4: In-Engine Editor Plugin

Build the vibe-coding interface as an editor plugin for each engine. The AI generates code + scene descriptions; the plugin, running inside the engine, materializes them into actual project assets using the engine's own APIs. This is the most powerful approach and avoids binary format issues entirely, but requires the developer to already have a project open.

EnginePlugin APINotes
Unreal UEditorUtilityWidget / Python scripting / EditorAssetLibrary UE5 supports Python scripting in the editor, which can create levels and assets
Unity AssetDatabase, EditorApplication, PrefabUtility Rich C# Editor API; can create any asset type programmatically
Godot EditorPlugin, EditorInterface, tool scripts Full editor access via GDScript @tool annotation; headless script execution also available

Strategy 5: The "Describe to Diff" Approach

Rather than generating complete project files from scratch, maintain a minimal base template for each engine and have the AI generate diffs/patches against it. The base template handles all binary stubs and boilerplate; the AI only touches the textual delta — scene structure, script logic, settings overrides. This is likely the most robust production approach.

Recommended Architecture
For a robust vibe-coding system: (1) Maintain versioned minimal base templates for each engine. (2) AI generates scene files, scripts, and config patches on top. (3) Headless engine invocation handles any remaining initialization. (4) The delivered output is a zip that opens cleanly in the target engine version.

The Recommended Pipeline Architecture

For a production-grade vibe-coding system that targets all three engines, the recommended approach is a layered template + AI delta model:

VibeCodePipeline:
  1. Parse user description / prompt
  2. Select engine target + version
  3. Load base template for engine/version from template store
     - Godot: ~10 files, all text
     - Unity: ~20 files, text + empty Assets/ skeleton
     - UE:    ~15 text files + 1 minimal .umap stub binary
  4. AI generates:
     - Scene/level description → engine scene format
     - System descriptions → script files
     - Settings overrides → config/settings files
  5. Copy GLB/glTF asset files into project (no pre-conversion needed)
     - Godot: drop into assets/ — native, zero config
     - Unity: drop into Assets/ — glTFast pkg already in manifest.json
     - UE (static):    drop into Content/ — Interchange handles import
     - UE (animated):  convert to FBX first via gltf-pipeline or Blender CLI
  6. Merge AI output over base template
  7. (Optional) Run headless engine to finalize cache + import assets
  8. Package and deliver project zip

The key insight across all three engines: the binary problem is largely a solvable initialization problem, not a deep architectural blocker. The engine itself is the best tool for generating the few binary files it needs — and all three engines support headless operation that can be piped into a generation workflow.

Bottom Line
Start with Godot to prototype the pipeline — it's the closest to "fully text, fully AI-writable" of any production game engine. Then apply the same architecture to Unity (requiring Force Text mode enforcement) and Unreal (requiring a stub .umap library + optional UBT invocation). The core generation logic — scene graph, script logic, settings — is engine-agnostic and can be translated between formats.
Section 08

glTF / GLB Asset Compatibility

The Babylon.js prototype layer of this system uses GLB/glTF as its native asset format — the same files that artists and procedural pipelines produce. The question is whether those same files can be handed directly to the three desktop engines, or whether a translation step is required before they become part of an exported template.

Key Architectural Difference
In Babylon.js, GLBs are loaded at runtime directly by the engine. In all three desktop engines, GLBs are imported at editor time and transcoded into the engine's internal native format — Unreal's .uasset, Unity's Library cache, Godot's .godot/imported/ cache. The source GLB persists in the project folder, but the engine renders from a processed copy. This means any data loss from unsupported material extensions or animation edge cases happens at that transcoding stage, not before.

Godot — Recommended Format, Zero Friction

Godot treats GLB as its preferred 3D interchange format, making it the ideal engine for a Babylon.js → Godot pipeline. You drop GLB files into the project directory and they become immediately importable as PackedScene resources. Bidirectional support — import from Blender/Babylon sources, export back out for round-tripping — has been present since Godot 4.0.

Godot Renderer Caveat
Rendering issues have been confirmed on the Vulkan Mobile and GL Compatible renderers (the ones targeting older hardware and mobile). The default Forward+ renderer handles glTF materials correctly. If your templates are intended for mobile targets, test the import pipeline against both renderers before shipping.

Godot also exposes its glTF data structures to GDScript, allowing custom import logic for proprietary glTF extensions — useful if your Babylon.js assets use extras or custom extension data that needs to survive the import into Godot.

Unity — One Package Declaration, Then Fully Functional

Unity has no native glTF support out of the box, but the official com.unity.cloud.gltfast package (originally the community com.atteneder.gltfast, now Unity-owned) resolves this completely. Adding it to Packages/manifest.json makes Unity register itself as the default importer for .gltf and .glb extensions, converting them to native Unity prefabs on import — no manual steps needed.

// Packages/manifest.json — add this to dependencies
{
  "dependencies": {
    "com.unity.cloud.gltfast": "6.9.1",
    "com.unity.inputsystem": "1.8.1",
    ...
  }
}

The material fidelity is excellent for the common PBR subset. The known limitations that matter for a template pipeline are:

LimitationImpactWorkaround
Max 2 UV sets Assets using 3+ UV channels lose the extra sets Merge UV data to 2 sets at export time

Build-Time Shader Gotcha
glTFast uses custom shader graphs that must be explicitly included in builds. If your template uses URP, add UnityGLTFShaderVariantCollection (or the BiRP equivalent) to the Preloaded Shaders list in Project Settings → Graphics. The template generator should write this into ProjectSettings/GraphicsSettings.asset automatically.

Unreal Engine — Static Assets Work, Animated Assets Need Care

Unreal Engine 5 includes a glTF importer via its Interchange Framework, which can be enabled with a single entry in the .uproject plugins array. Static environment assets, props, and PBR materials import correctly and reliably.

// MyProject.uproject — enable Interchange glTF support
{
  "Plugins": [
    { "Name": "InterchangeEditor", "Enabled": true },
    { "Name": "GLTFImporter",      "Enabled": true }
  ]
}

The meaningful friction is with skeletal meshes and animations. Community reports across multiple UE5.x versions document persistent issues: skeletal animations from GLBs can exhibit stretching, incorrect bone scales, and transform mismatches that don't appear in other glTF viewers or in Godot/Unity. Static assets from the same GLB files import without issue.

Animated Character Assets — Use FBX for Unreal
For any template that includes rigged, animated characters, the safest approach for the Unreal export path is to convert to FBX at pipeline time rather than rely on the glTF animation importer. This is a one-command operation using freely available tooling and can be automated entirely within the export pipeline. Static meshes and environments can remain as GLB/glTF.

Automated Conversion for the Unreal Animation Path

The conversion from GLB → FBX can be fully automated using either Blender's CLI (free, open source) or the Khronos gltf-pipeline tool combined with a converter. Blender is the most reliable path for preserving animation fidelity:

#!/usr/bin/env python3
# convert_animated_glb_to_fbx.py
# Runs headless via: blender --background --python this_script.py -- input.glb output.fbx

import bpy, sys

argv = sys.argv[sys.argv.index("--") + 1:]
input_glb, output_fbx = argv[0], argv[1]

bpy.ops.wm.read_factory_settings(use_empty=True)
bpy.ops.import_scene.gltf(filepath=input_glb)
bpy.ops.export_scene.fbx(
    filepath=output_fbx,
    use_selection=False,
    add_leaf_bones=False,
    bake_anim=True,
    bake_anim_use_all_actions=True,
    path_mode='COPY',
    embed_textures=True
)

# Shell invocation:
# blender --background --python convert_animated_glb_to_fbx.py -- character.glb character.fbx

This script runs entirely headlessly — no Blender GUI is opened. It can be integrated as a conditional step in the template export pipeline: if the target is Unreal and the asset contains animation data, run the conversion; otherwise, pass the GLB through unchanged.

Pipeline Decision Logic

Putting it together, the asset handling decision for each export target is straightforward:

for each GLB asset in template:
  if target == "godot":
    copy GLB as-is → project/assets/
    # Godot imports natively, no action needed

  elif target == "unity":
    copy GLB as-is → project/Assets/models/
    # glTFast (in manifest.json) handles the rest
    # Flag assets with 3+ UV sets or transmission for manual review

  elif target == "unreal":
    if asset.has_skeletal_animation:
      run blender_convert(asset, format="FBX") → project/Content/Characters/
    else:
      copy GLB as-is → project/Content/Models/
      # Interchange imports static GLBs correctly
GLB Compatibility Summary
No pre-conversion is needed for Godot or Unity — GLB files drop directly into both engines without any pipeline transformation. For Unreal, static environment and prop assets are equally straightforward; the only case requiring a conversion step is skeletal/animated character assets destined for UE, where a headless Blender FBX conversion is the reliable solution. Since Babylon.js is the Khronos-standard glTF runtime par excellence, assets authored for it are well-positioned for cross-engine use with minimal fidelity loss.
